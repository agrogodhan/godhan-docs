| — | **A second serious, platform-wide bug found and fixed, this time entirely on the mobile side**: every non-auth API service (`CattleApiService`, `CattleMarketApiService`, `DairyShopApiService`, `DashboardApiService`, `WalletApiService`, `ReportsApiService`, `NotificationsApiService`, `ReferralApiService`) deserialized raw HTTP responses directly into domain types, never unwrapping the `{ success, message, data }` envelope every backend controller actually sends. For list-returning calls (e.g. the herd list) this **throws on every call** — an object where an array is expected. For object-returning calls it's worse: since most domain types have all-default fields, deserialization "succeeds" silently and produces an empty/garbage object with no error surfaced anywhere. Only `AuthApiService` did this correctly (built that way from the start). Fixed by adding one shared `HttpResponse.unwrap()`/`unwrapVoid()`/`unwrapMessage()` extension (matching the pattern `AuthApiService` already had) and switching all 9 files onto it, `AuthApiService` included, so there's one implementation instead of two.
